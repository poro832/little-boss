# LittleBoss 프론트엔드-백엔드 연결 가이드

## 1. 현재 상태 요약

### 프론트엔드 (React + Vite)
- **레포**: https://github.com/aosxou/2026_CapstoneProject
- **배포**: Vercel (개발용) → S3 정적 호스팅 (프로덕션)
- **구조**: `LittleBoss.jsx` 단일 파일에 모든 컴포넌트 포함
- **데이터**: 모든 데이터가 하드코딩 또는 `localStorage` → **API 연결 전무**
- **인증**: 로그인/회원가입 UI만 존재, 실제 인증 로직 없음

### 백엔드 (AWS 비동기 파이프라인)
- **로컬**: Flask `local_server.py` (`http://localhost:5000`) — 동기 처리
- **프로덕션**: API Gateway + Lambda 4개 — **비동기 이벤트 드리븐**
- **AI**: Gemini API (`google-generativeai`)
- **인증**: 미구현 (`user_id` 파라미터로 대체)

### 핵심 차이: 로컬 vs AWS 흐름

```
[로컬 — 동기]
POST /process → 업로드 + OCR + AI 분석 한번에 → 결과 즉시 반환

[AWS — 비동기]
POST /upload → doc_id 즉시 반환 → OCR 자동 → AI 자동 → 프론트에서 폴링
```

> ⚠️ AWS 배포 시 `/process`, `/ocr/{doc_id}`, `/analyze/{doc_id}` 엔드포인트는 존재하지 않습니다.
> OCR과 AI 분석은 S3 이벤트 / DynamoDB Streams 트리거로 자동 실행됩니다.

## 2. API 엔드포인트 (AWS 기준)

aws_setup.md의 API Gateway 설정 기준입니다.

| 메서드 | 경로 | Lambda | 프론트엔드 페이지 | 설명 |
|--------|------|--------|------------------|------|
| GET | `/health` | upload-handler | - | 서버 상태 확인 |
| POST | `/upload` | upload-handler | `UploadPage` | 파일 업로드 → **doc_id 즉시 반환** |
| GET | `/documents` | upload-handler | `Dashboard`, 문서 목록 | 사용자 문서 목록 |
| GET | `/documents/{doc_id}` | upload-handler | `DocumentAnalysisPage` | **상태 폴링** (프론트엔드 핵심) |
| POST | `/calendar/{doc_id}` | action-executor | `SchedulePage` | 캘린더 등록 (사용자 클릭) |
| GET | `/checklist/{doc_id}` | action-executor | `DocumentDetailPage` | 체크리스트 조회 |
| PATCH | `/checklist/{doc_id}` | action-executor | `DocumentDetailPage` | 체크 완료 처리 |

> `/ocr`, `/analyze`, `/process` 는 AWS에 없음 — 비동기 트리거로 자동 실행

## 3. 비동기 파이프라인과 프론트엔드 폴링

### 전체 흐름

```
[프론트엔드]
  │ POST /upload (파일 전송)
  ▼
[API Gateway → Lambda: upload-handler]
  │ 1. S3에 파일 저장 (uploads/{doc_id}/파일명)
  │ 2. DynamoDB 레코드 생성 (status: "uploaded")
  │ 3. 즉시 응답 → { doc_id, status: "uploaded" }
  │
  ▼ (S3 이벤트 트리거 — 자동, 비동기)
[Lambda: ocr-handler]
  │ Textract OCR → DynamoDB status → "ocr_done"
  │
  ▼ (DynamoDB Streams 트리거 — 자동, 비동기)
[Lambda: ai-analyzer]
  │ Gemini API 호출 → 분석 결과 저장 → status → "done"

[프론트엔드] ← GET /documents/{doc_id} 를 3초 간격 폴링
  │ status === "done" → 결과 화면 표시
  │ status === "error" → 에러 메시지 표시
```

### 상태 흐름 (DynamoDB status 필드)

```
uploaded → ocr_done → ai_processing → done
                ↘                ↘
              error              error
```


```js
import axios from 'axios';

const api = axios.create({
  baseURL: import.meta.env.VITE_API_URL || 'http://localhost:5000',
  timeout: 30000,
});

export default api;
```

```js
  }
```

### 4-2. 페이지별 연결 작업

#### UploadPage — 파일 업로드 + 폴링 (핵심, 최우선)

**현재**: 파일 선택 시 가짜 progress bar만 표시, 서버 전송 없음

**변경할 것 (AWS 비동기 방식)**:
```js
const addFiles = async (files) => {
  for (const file of files) {
    const formData = new FormData();
    formData.append('file', file);
    formData.append('user_id', userId);

    const item = { name: file.name, progress: 0, id: Date.now() + file.name };
    setQueue(q => [...q, item]);

    try {
      // 1. 업로드 → doc_id 즉시 반환
      const res = await api.post('/upload', formData, {
        onUploadProgress: (e) => {
          const pct = Math.round((e.loaded * 100) / e.total);
          setQueue(q => q.map(i => i.id === item.id ? { ...i, progress: pct } : i));
        }
      });

      if (res.data.doc_id) {
        // 2. 폴링 시작 — OCR + AI 분석 완료까지 대기
        pollDocumentStatus(res.data.doc_id);
      }
    } catch (err) {
      console.error('업로드 실패:', err);
    }
  }
};

// 3초 간격 폴링
const pollDocumentStatus = (docId) => {
  const interval = setInterval(async () => {
    try {
      const res = await api.get(`/documents/${docId}`, {
        params: { user_id: userId }
      });
      const status = res.data.status || res.data.document?.status;

      if (status === 'done') {
        clearInterval(interval);
        onNavTo('doc-analysis', docId); // 결과 페이지로 이동
      } else if (status === 'error') {
        clearInterval(interval);
        toast('분석 중 오류가 발생했습니다');
      }
      // uploaded, ocr_done, ai_processing → 계속 폴링
    } catch (err) {
      clearInterval(interval);
      console.error('폴링 실패:', err);
    }
  }, 3000);
};
```

> **로컬 개발 시**: Flask의 `POST /process` (동기)를 사용해도 됨. 단, AWS 배포 전 반드시 폴링 방식으로 전환 필요.

## 7. 주의사항

1. **로컬 ↔ AWS 흐름 차이**: 로컬(`/process` 동기)과 AWS(`/upload` + 폴링 비동기)는 흐름이 다름. 로컬 개발 시에도 폴링 방식으로 작성하면 AWS 전환이 수월함
2. **백엔드 응답 형식 통일**: 프론트에서 기대하는 데이터 구조와 백엔드 응답이 다를 수 있음 → 연결 시 응답 매핑 작업 필요
3. **AI 분석 시간**: 업로드 후 OCR + Gemini 분석까지 10~30초 소요 → 상태별 로딩 UI 필수
4. **파일 크기 제한**: 백엔드 10MB, 프론트엔드 UI에는 20MB라 표시 → 10MB로 통일 필요
5. **CORS**: 로컬 개발 시 Vite 프록시 사용, AWS 배포 시 API Gateway CORS 설정 (`littleboss-frontend` S3 도메인만 허용)
6. **`localStorage` 잔존 코드**: 체크리스트 API 연결 후 `localStorage` 코드 정리 필요
7. **Access Key 미사용**: AWS CLI는 CloudShell 또는 EC2에서 IAM Role 기반으로만 사용 (키 발급 금지)
